Introducción

PanaLogis resuelve un problema operativo común en empresas pequeñas y medianas de transporte: la información crítica del negocio suele dispersarse entre hojas de cálculo, chats, llamadas y memoria del personal. Esa forma de trabajo provoca asignaciones duplicadas, falta de trazabilidad, retrasos de cobro y baja visibilidad sobre flota, conductores y órdenes.

El proyecto plantea una base de datos relacional como núcleo del sistema y una aplicación web ligera como capa operativa. La decisión de diseño central fue ubicar la lógica de negocio crítica en la base de datos, no en la interfaz. Por eso el sistema bloquea conductores ocupados, evita usar vehículos en mantenimiento, genera facturas al entregar órdenes y registra cambios en bitácora mediante triggers y procedimientos almacenados.

El resultado cubre el recorrido completo del cuatrimestre: diagnóstico, análisis, modelado conceptual, modelo relacional, normalización, implementación SQL, consultas funcionales, documentación técnica y verificación operativa. El script académico oficial se mantiene en MariaDB bajo database/panalogis.sql, mientras que el despliegue cloud usa un adaptador PostgreSQL solo para demostración pública.

I. Planteamiento del problema

Problemática identificada

En una operación de transporte de carga, la asignación de rutas, vehículos y conductores exige consistencia inmediata. Si la empresa trabaja con registros manuales, el operador puede asignar el mismo conductor a dos servicios, despachar un camión que está en taller o perder el seguimiento de facturas pendientes. Cada error impacta servicio, costos y credibilidad frente al cliente.

El problema no es solo tecnológico. También es de control. Cuando la lógica del negocio vive únicamente en la cabeza del operador, la empresa depende de memoria humana y de revisiones tardías. Eso hace imposible garantizar integridad operativa si cambian turnos, crece la carga de trabajo o se requiere auditar lo ocurrido.

Antecedentes de la problemática identificada

El curso de Base de Datos I exigía resolver una necesidad real o simulada del entorno institucional o laboral. El equipo escogió el caso de una empresa de transporte terrestre porque combina inventario móvil, personal operativo, documentos comerciales y eventos con fuerte dependencia temporal. En este contexto, una base de datos relacional permite modelar entidades claras, reglas de integridad y trazabilidad de estados.

A partir del análisis del flujo de trabajo típico del sector, se identificaron cuatro dolores principales: falta de visibilidad integral, retrasos de cobro, mantenimiento mal planificado y ausencia de auditoría sobre cambios de estado. PanaLogis se diseñó para atacar esos puntos con un modelo de datos normalizado y reglas de negocio ejecutadas desde SQL.

II. Análisis

Propuesta de solución

La solución consiste en un sistema web conectado a una base de datos relacional centralizada. El operador registra flota, conductores, clientes, rutas, tipos de carga y órdenes. La base valida reglas críticas y automatiza eventos derivados. La interfaz no reemplaza la lógica; la expone de forma controlada.

Decisión de arquitectura: la base de datos es el punto único de verdad. La aplicación Flask sirve para operar, consultar y visualizar, pero la integridad vive en triggers, llaves foráneas, restricciones y procedimientos almacenados.

Justificación

El modelo relacional ofrece una ventaja directa para este caso: separa entidades, evita duplicidad y garantiza relaciones consistentes. Esa estructura era necesaria para un escenario donde un cliente puede generar muchas órdenes, un vehículo puede pasar por varios mantenimientos y una orden entregada debe convertirse en factura solo una vez.

Objetivo del proyecto

Objetivo general

Diseñar, implementar y documentar un prototipo funcional de base de datos relacional que permita gestionar de forma íntegra la operación de una empresa de transporte de carga terrestre.

Objetivos específicos

  • Modelar las entidades y relaciones críticas del negocio mediante DER y modelo relacional.
  • Normalizar el esquema hasta tercera forma normal en las tablas operativas principales.
  • Implementar triggers y procedimientos almacenados para reglas de negocio repetibles.
  • Construir una interfaz web que permita ejecutar el flujo operativo completo.
  • Validar el prototipo con consultas funcionales, pruebas automatizadas y evidencia visual.

Metodología utilizada

  1. Diagnóstico: definición del problema y del alcance logístico.
  2. Diseño conceptual: identificación de entidades, atributos y cardinalidades.
  3. Diseño lógico: transformación del DER en tablas con llaves primarias y foráneas.
  4. Implementación: construcción del script SQL, la aplicación Flask y la capa de reportes.
  5. Verificación: carga de datos de prueba, consultas, pruebas automatizadas y documentación.

Restricciones

  • Tiempo académico limitado al cuatrimestre.
  • Uso obligatorio de SQL relacional y documentación formal.
  • Stack ligero sin ORM para mantener control explícito sobre consultas.
  • Hospedaje gratuito para la demo pública.

Limitantes y delimitantes

  • No se implementó autenticación completa por roles en la interfaz final.
  • El alcance se concentró en operación terrestre de carga.
  • La facturación cubre el flujo operativo base, no contabilidad avanzada.
  • La nube se utilizó como canal de demostración, no como cambio del fundamento académico.

III. Diseño

Diagrama Entidad-Relación (DER)

El DER identifica tres grupos: catálogos maestros, operación transaccional y soporte administrativo. La tabla central es ORDEN_SERVICIO, porque conecta cliente, ruta, vehículo, conductor y tipo de carga. A partir de ella nacen los procesos de facturación y bitácora.

Catálogos maestros
  • TIPO_VEHICULO define clase y capacidad.
  • TIPO_CARGA clasifica mercancía y banderas especiales.
  • RUTA consolida origen, destino, distancia y tarifa base.
  • ROL_USUARIO define perfiles administrativos.
Entidades operativas
  • VEHICULO representa la flota física.
  • CONDUCTOR controla disponibilidad y licencias.
  • CLIENTE concentra datos comerciales.
  • ORDEN_SERVICIO registra el evento logístico principal.
Entidades derivadas
  • MANTENIMIENTO vincula eventos técnicos con vehículo.
  • FACTURA nace automáticamente al entregar una orden.
  • BITACORA guarda trazabilidad de cambios de estado.
Soporte administrativo
  • USUARIO relaciona identidad de sistema con rol.
  • Las referencias permiten escalar a seguridad y auditoría futura.

Relaciones y cardinalidades

RelaciónCardinalidadRegla de negocio
TIPO_VEHICULO → VEHICULO1 : NUn tipo puede clasificar muchos vehículos; cada vehículo tiene un solo tipo.
CLIENTE → ORDEN_SERVICIO1 : NUn cliente puede generar múltiples órdenes.
RUTA → ORDEN_SERVICIO1 : NLa orden toma una ruta predefinida con tarifa base.
VEHICULO → ORDEN_SERVICIO1 : NUn vehículo participa en muchas órdenes a lo largo del tiempo, pero no en dos activas al mismo tiempo.
CONDUCTOR → ORDEN_SERVICIO1 : NUn conductor puede atender varias órdenes históricas, pero no dos activas simultáneas.
ORDEN_SERVICIO → FACTURA1 : 1Una orden entregada genera una sola factura.
VEHICULO → MANTENIMIENTO1 : NUn vehículo acumula historial de mantenimiento.

Conversión del DER al modelo relacional

Cada entidad del DER se transformó en tabla con clave primaria numérica autoincremental. Las relaciones uno a muchos se materializaron mediante claves foráneas y la relación uno a uno entre orden y factura se aseguró con restricción UNIQUE sobre FACTURA.id_orden. Los tipos de datos se asignaron según naturaleza del dato: identificadores enteros, textos variables, fechas, montos decimales y estados como ENUM.

CREATE TABLE ORDEN_SERVICIO (
  id_orden         INT AUTO_INCREMENT PRIMARY KEY,

  id_cliente       INT NOT NULL,
  id_ruta          INT NOT NULL,
  id_vehiculo      INT NOT NULL,

Pruebas de reglas de negocio

  • La creación de órdenes persiste observaciones y bloquea conductor o vehículo no disponibles.
  • Al cambiar una orden a ENTREGADO, se genera su factura correspondiente.
  • Al registrar mantenimiento en proceso, el vehículo cambia a MANTENIMIENTO; al completarlo, vuelve a ACTIVO.
  • Las facturas anuladas se excluyen de reportes financieros y de rentabilidad.

VI. Conclusiones

La conclusión principal del proyecto es que una base de datos bien diseñada puede asumir reglas de negocio críticas sin depender del código de interfaz. En PanaLogis, la consistencia operativa se sostuvo gracias al modelo relacional, las restricciones y la automatización SQL. Esa decisión reduce errores humanos y facilita auditoría.

El segundo hallazgo es que el modelado previo sí impacta la calidad final. La separación de catálogos, eventos y documentos derivados permitió mantener el sistema en tercera forma normal y, al mismo tiempo, construir reportes útiles sin duplicidad de datos.

El prototipo quedó en un punto funcional real: puede demostrarse localmente, puede navegarse en la nube y su comportamiento ya fue validado con pruebas repetibles. Para un proyecto de Base de Datos I, eso demuestra dominio de diseño, implementación y documentación.

VII. Desafíos encontrados

De la propuesta final con el proceso de enseñanza-aprendizaje

El desafío técnico más fuerte fue mantener coherencia entre lo académico y lo operativo. El curso exigía una base relacional sólida en MariaDB. La demostración pública, por costo y disponibilidad, exigía hosting gratuito. El equipo resolvió esa tensión manteniendo el script académico intacto y creando un adaptador cloud sin alterar el fundamento del modelo.

También surgieron retos de integración entre base de datos y aplicación. Las diferencias entre cursores, manejo de procedimientos almacenados y despliegue serverless obligaron a revisar con rigor cada flujo en vez de asumir que el código funcionaría igual en todos los entornos.

Tus aportes al curso

  • Martín Bundy lideró la integración general, el despliegue y la consolidación técnica del producto.
  • Miguel Herrera Briceño aportó enfoque de QA y análisis SQL para validar reglas de negocio y reportes.
  • Jesús De León sostuvo la línea documental y la preparación de la sustentación oral.

Recomendaciones enfocadas a cada conclusión brindada

  • Mantener las reglas críticas en la base de datos siempre que el proceso dependa de integridad transaccional.
  • Versionar formalmente el script SQL desde el inicio del proyecto para reducir ajustes tardíos.
  • Ampliar en una versión futura el módulo de autenticación y control granular por rol.
  • Agregar indicadores de mantenimiento preventivo y vencimiento de licencias como alertas proactivas.

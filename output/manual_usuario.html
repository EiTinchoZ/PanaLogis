1. Alcance del manual

Este manual explica cómo usar el sistema real que quedó desplegado y probado. Está pensado para operador, profesor o compañero que necesite abrir la aplicación, comprender el flujo y ejecutar una demostración breve sin revisar el código.

Estado verificado: el sistema respondió correctamente en producción y la suite local cerró en 12 de 12 pruebas exitosas el 9 de abril de 2026.

2. Cómo ingresar al sistema

Opción A. Usar la demo pública

  1. 1
    Abrir el navegador.
  2. 2
    Entrar a https://panalogis.vercel.app.
  3. 3
    El sistema redirige al dashboard automáticamente.

Opción B. Usar el entorno local

  1. 1
    Iniciar XAMPP y confirmar que MariaDB está arriba.
  2. 2
    Importar database/panalogis.sql si la base no existe.
  3. 3
    Ejecutar python app.py desde la carpeta del proyecto.
  4. 4
    Abrir http://127.0.0.1:5000.
Importante: para la sustentación conviene usar primero la demo pública. Si el internet falla, el plan B es correr la versión local.

3. Navegación general

La barra lateral izquierda organiza los módulos. El panel superior muestra el título de la vista actual. Las tablas usan acciones directas para ver detalle, editar o regresar al listado. El dashboard es el punto de entrada y resume el estado del turno.

ZonaUso
SidebarEntrar a cualquier módulo del sistema.
TopbarUbicar la vista actual y contexto de navegación.
TablasConsultar registros, abrir detalles y ejecutar acciones.
FormulariosRegistrar o editar datos operativos.
Mensajes flashConfirmar éxito, advertencia o error después de una acción.

4. Dashboard

El dashboard muestra el resumen del negocio en una sola vista: vehículos activos, órdenes en tránsito, conductores disponibles, unidades en mantenimiento, órdenes recientes, radar financiero, rutas calientes y bitácora. También incluye un copiloto IA para consultas rápidas del turno.

5. Operación por módulo

Flota

  • Permite listar, registrar, ver y editar vehículos.
  • Cada vehículo guarda placa, marca, modelo, año, tipo y kilometraje.
  • El estado puede ser ACTIVO, MANTENIMIENTO o INACTIVO.

Conductores

  • Registra cédula, licencia, categoría y fecha de vencimiento.
  • La disponibilidad real no depende del formulario: la base bloquea asignaciones indebidas.

Clientes

  • Centraliza RUC, razón social y datos de contacto.
  • Sirve como origen del historial comercial en reportes.

Órdenes

  • Es el módulo central del sistema.
  • Usa el procedimiento almacenado sp_crear_orden.
  • Los estados operativos son PENDIENTE, EN_TRANSITO, ENTREGADO y CANCELADO.

Mantenimiento

  • Al abrir un mantenimiento en proceso, el vehículo cambia a estado de taller.
  • Al completar el mantenimiento, el vehículo vuelve a activo.

Facturas

  • Las facturas no se crean manualmente. Nacen cuando una orden se entrega.
  • Se pueden actualizar a PAGADA o ANULADA.

Reportes

  • Agrupan 8 consultas funcionales del proyecto.
  • Son la mejor evidencia para sustentar la utilidad del modelo relacional.

6. Flujos recomendados para la sustentación

Flujo 1. Mostrar la salud operativa

  1. 1
    Abrir el dashboard.
  2. 2
    Explicar KPIs, radar financiero y bitácora.
  3. 3
    Entrar a reportes y enseñar la rentabilidad por ruta.

Flujo 2. Mostrar una regla de negocio

  1. 1
    Abrir Órdenes.
  2. 2
    Explicar que no se puede usar un conductor ocupado o un vehículo en mantenimiento.
  3. 3
    Cambiar una orden a entregada y mencionar que la factura se genera por trigger.

Flujo 3. Mostrar automatización técnica

  1. 1
    Abrir Mantenimiento.
  2. 2
    Explicar que el estado del vehículo cambia desde la base de datos.
  3. 3
    Relacionar ese cambio con la integridad del sistema.

7. Reglas de negocio en lenguaje simple

ReglaQué significa para el usuario
Conductor no disponibleNo puedes asignarlo a una orden nueva si ya tiene una activa.
Vehículo en mantenimientoNo puede participar en una orden mientras esté en taller.
Factura automáticaLa factura aparece cuando la orden cambia a entregada.
BitácoraLos cambios de estado quedan registrados para auditoría.
Facturas anuladasNo se suman en los reportes financieros.

8. Mensajes frecuentes y cómo responder

“El conductor ya tiene una orden activa asignada.” Debes escoger otro conductor o cerrar la orden anterior.
“El vehículo está en mantenimiento y no puede operar.” Revisa el módulo de mantenimiento y usa otra unidad.
“Confirma la acción.” El sistema protege cambios sensibles como anulaciones o eliminaciones.
Mensaje de éxito. Confirma que el registro quedó guardado o actualizado sin errores.

9. Recomendaciones finales de uso

  • Para la exposición, abre primero la demo pública y deja una pestaña local lista como respaldo.
  • Explica siempre que la lógica crítica vive en la base de datos, no solo en la interfaz.
  • Si necesitas demostrar valor académico, entra a reportes y conecta cada resultado con una consulta funcional del informe.
  • Si el profesor pregunta por portabilidad, aclara que MariaDB es la base oficial y que Vercel/Supabase solo se usó para hospedar la demo.
